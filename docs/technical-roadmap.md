# LAS 点云 Web Viewer 技术路线

本文面向熟悉 C++/Qt/OpenGL、但不熟悉 Web 前端的开发者，说明当前产品的技术架构、LAS 加载链路、效率优化方式，以及后续升级路线。产品定位是浏览器端 LAS 点云查看、航线/部件展示编辑、简单点云分析和电力巡检辅助，不规划 LAZ 支持。

## 产品定位

这个项目不是一个通用文件预览页，而是面向三维点云电力巡检场景的 Web 工作台。短期目标是让用户能顺畅打开 LAS 点云、航线 JSON、部件点和航点，完成浏览、定位、测量、剖面、净空初筛和航线编辑；中期目标是接近市面常见点云软件的交互体验；长期目标是把输电、变电场景的专用分析和巡检结果管理补齐。

产品升级优先级应遵循三个原则：

- 先把浏览和交互做顺，再扩展复杂分析。
- 先保证 LAS 本地加载和 WebGL/Web Worker 稳定，再考虑 WASM 或 WebGPU。
- 先做贴合电力巡检的高频工具，例如航点、部件点、导线、杆塔、净空、交跨和问题点，再做泛协同能力。

## 当前技术栈

- UI 层：HTML、CSS、JavaScript。
- 构建与开发服务：Node.js、Vinext/Vite。
- 点云渲染：WebGL，对应 Qt 中的 `QOpenGLWidget` 或自管 OpenGL 渲染窗口。
- 本地文件读取：浏览器 File API，LAS 文件在本机浏览器内解析，不上传服务器。
- 二进制数据结构：`ArrayBuffer`、`DataView`、`Float32Array`、`Uint8Array`，对应 C++ 中连续内存 buffer。
- 业务模块：航线 JSON、部件点、航点编辑、净空分析、标注和测量均以浏览器端 JavaScript 模块实现。
- 项目管理：支持多个 LAS 点云和多个航线 JSON 加载到项目目录树，通过显隐、激活、定位和删除管理图层。

## 与 C++/Qt 的类比

- `public/viewer/index.html`：类似主窗口 UI 布局。
- `public/viewer/src/main.js`：类似 MainWindow/controller，负责文件加载、状态管理、UI 事件和模块协调。
- `public/viewer/src/las.js`：类似 LAS reader，用 `DataView` 解析二进制 header 和 point records。
- `public/viewer/src/renderer.js`：类似 OpenGL renderer，维护相机、VBO、shader、拾取和绘制。
- `public/viewer/src/color.js`：类似颜色映射与过滤模块，生成用于 GPU 上传的颜色 buffer。
- `public/viewer/src/route.js`：类似航线数据模型和 JSON 适配层。
- `requestAnimationFrame`：类似 Qt 的 `update()`/paint 调度，但由浏览器按刷新节奏执行。

## LAS 加载链路

1. 用户选择或拖入 `.las` 文件。
2. 浏览器通过 `file.arrayBuffer()` 将本地文件读入内存。
3. `las.js` 读取 LAS header，确认 point format、点数、offset、scale、RGB、classification 等字段。
4. 根据点数和浏览器预算计算抽样步长 `stride`。
5. 将点坐标转换到局部坐标系，并写入 `Float32Array`。
6. 将 RGB 或分类/高程颜色写入 `Uint8Array`。
7. `renderer.setCloud()` 将坐标和颜色上传到 WebGL buffer。
8. WebGL 通过 `gl.drawArrays(gl.POINTS, 0, pointCount)` 批量绘制点云。

当前路线是浏览器本地解析 LAS，不依赖后端服务，不支持 LAZ 压缩格式。

多 LAS 场景下，第一版以首个点云的 LAS 中心作为项目局部坐标原点。后续加载的 LAS 会从各自局部坐标转换到同一项目局部坐标系，再把所有可见点云合成为一个 WebGL 渲染 buffer。这个方案能尽快提供目录树和多图层工作流；后续大规模数据再升级为真正的多 buffer、分块和 LOD。

## 当前效率优化

- 点数抽样：设置最大渲染点数预算，超大 LAS 按 `ceil(pointCount / maxRenderPoints)` 抽样，避免一次性绘制几千万点导致浏览器卡死。
- TypedArray 连续内存：坐标和颜色使用 typed array，减少对象分配和 GC 压力，便于直接上传 GPU。
- GPU buffer 渲染：点云不是 DOM 节点，而是 WebGL VBO 批量渲染。
- 局部坐标：点云坐标以中心点归一到局部空间，降低大坐标对浮点精度的影响。
- 按需渲染：相机变化、显示模式变化、窗口 resize、数据更新时才请求下一帧，避免空闲时持续占用 CPU/GPU。
- 颜色重建分离：RGB、高程、类别、过滤和裁剪在 `color.js` 中统一生成渲染 buffer，保持渲染器简单。
- 可见图层合成：项目树中可见的 LAS 图层会分别完成分类/高程/RGB 着色和过滤，再合成为一个连续 GPU buffer。
- 轻量分析：剖面、净空和相机预览基于当前抽样点集计算，保证第一版交互流畅。

## 只支持 LAS 的边界

短中期只支持 `.las`，不支持 `.laz`。原因：

- LAS 是未压缩点记录，浏览器端二进制解析直接、可控。
- LAZ 需要解压库和额外 CPU/WASM 成本，会明显增加工程复杂度。
- 当前产品优先验证电力巡检工作流、航线编辑、显示和基础分析能力，LAS 已满足第一阶段数据入口。

如果后续确实需要接入 LAZ，应作为独立里程碑评估，不混入当前 LAS 渲染优化路线。

## 现代点云产品体验目标

参考 Potree、CesiumJS、Pix4Dcloud、NavVis IVION、Trimble Connect 等产品，当前项目最值得补齐的是低耦合、高感知、强反馈的工作台体验，而不是过早做账号、云协作或复杂服务端切片。

体验目标分为六类：

- 浏览更顺：左键旋转、中键/右键平移、滚轮缩放、漫游模式、快捷视角、视角小组件、视图书签都要稳定可预期。
- 画面更清楚：EDL、点大小衰减、分类透明度、背景切换、选中高亮、视椎体和标签层级要能帮助用户看清导线、杆塔、植被、地面和交跨。
- 信息更可管：目录树、图层显隐、活动图层、文件状态、渲染点数、抽样比例、性能状态需要可见，不能让用户猜当前软件在做什么。
- 编辑更像专业软件：航点和部件点列表要清晰，双击编辑、表格编辑、右侧属性面板、视椎体预览、局部相机预览和撤销恢复要逐步补齐。
- 分析更贴近电力：分类色表、导线/地线/杆塔/绝缘子/植被/交跨专题显示、净空风险、剖面和问题点要围绕巡检流程组织。
- UI 更现代：界面应采用稳定的工作台布局、清晰层级、紧凑工具栏、可折叠面板、统一颜色令牌和更少干扰的视觉装饰。

## 目标工作台形态

推荐把 Viewer 演进为“四区一浮层”的现代点云工作台：

- 顶部工具栏：文件打开、显示模式、测量/剖面/净空、截图、重置视图、性能状态、全局搜索或命令入口。
- 左侧项目树：点云、航线、标注、分析结果分组管理，支持显隐、激活、定位、删除和图层状态提示。
- 中央三维视图：WebGL 点云主画布，叠加航线、航点、部件点、标签、视椎体、测量线、剖面线和问题点。
- 右侧属性面板：按“显示/分析/航点/部件点/航线 QA/设置”分 tab，避免侧边栏无限滚动。
- 浮层窗口：航点相机预览、剖面图、编辑弹窗、快捷视角、性能小窗等需要随场景上下文出现的面板。

这个布局更接近桌面点云软件，也更适合从 Qt 工作流迁移过来的用户。

## 升级路线

### 阶段 0：当前基线

当前版本已经具备：

- LAS 本地加载、抽样、RGB/高程/类别显示。
- 默认电力场景分类色表与类别显隐。
- EDL、点大小、截图、视角小组件、快捷视角。
- 测量、剖面、净空辅助分析。
- 航线 JSON、部件点、航点、视椎体、相机预览和基础编辑。
- 多 LAS 与多航线目录树，支持显隐、激活、定位和删除。

这一阶段的价值是验证浏览器端点云巡检工具的可行性。

### 阶段 1：浏览与交互手感

目标是让用户觉得“能用、顺手、像专业软件”。

- 统一相机控制模型：Orbit、Pan、Zoom、Walk、Fit、Focus 都通过同一套相机状态管理。
- 补齐鼠标习惯设置：旋转水平/垂直反转、平移水平/垂直反转、滚轮缩放方向、旋转速度、平移速度、缩放速度。
- 强化快捷视角：俯视、前视、侧视、等轴测、沿航线观察、选中对象聚焦、上一视角/下一视角。
- 增加视图书签：保存相机位置、目标点、显示模式、激活图层和备注，便于巡检复盘。
- 完善拾取反馈：点、航点、部件点、问题点 hover 和 selected 状态要有清楚的颜色、标签和属性反馈。
- 建立操作撤销：航点移动、姿态修改、属性编辑、显隐切换至少应支持基础撤销/恢复。

### 阶段 2：现代 UI 与信息架构

目标是减少侧边栏拥挤，让用户用更少滚动完成高频操作。

- 将右侧面板模块化：显示、项目、航点、部件点、分析、设置分成 tab 或可停靠面板。
- 项目树升级为图层管理器：图层类型图标、渲染点数、抽样比例、错误状态、锁定、透明度和颜色入口。
- 航点/部件点列表升级为专业表格：固定表头、列宽调整、筛选、排序、选中同步、双击编辑、批量显隐。
- 编辑窗口升级为属性编辑器：坐标、机头偏航、云台俯仰、焦距倍率、相机偏航/俯仰、关联部件、关联目标统一展示。
- 做好空状态和错误状态：未加载、解析中、图层隐藏、数据不匹配、WebGL 不可用、文件过大都要有明确提示。
- 建立视觉规范：深色三维视图区、浅色或中性色属性面板、稳定强调色、统一按钮/表格/标签/弹窗样式。

### 阶段 3：渲染质量与视觉表达

目标是让用户看得清复杂输电场景。

- EDL 默认调优：提供轻/中/强预设，避免用户只看到“开关”却不知道怎么调。
- 点大小策略升级：支持固定点大小、距离衰减、按分类点大小、按选中对象强调。
- 分类专题显示：导线、地线、杆塔、植被、建筑、交跨、绝缘子等类别可一键独显、半透明、锁定或高亮。
- 标签层级控制：远处显示编号，近处显示名称和关键属性，避免大量标签遮挡。
- 视椎体视觉统一：普通、hover、selected、编辑中采用不同颜色和线宽，选中航点联动右下角相机预览。
- 多路线表达：至少支持活动航线重点显示，后续支持多个可见航线同时低透明展示。

### 阶段 4：性能管线升级

目标是从“抽样能看”升级到“大数据仍然流畅”。

- Web Worker 解析：LAS header、点记录读取、颜色重建、剖面和净空等重计算移出主线程。
- Transferable typed array：Worker 与主线程之间转移 buffer 所有权，减少大数组拷贝。
- 渐进加载：先显示粗抽样全局，再逐步补充更密点集，避免用户长时间等待空画面。
- 多 buffer 图层：每个 LAS 图层独立 WebGL buffer，替代当前合成单 buffer，支持独立显隐、透明度和更新。
- 分块与 LOD：按空间网格或八叉树组织 LAS 点，结合视锥裁剪、距离裁剪和屏幕误差控制点预算。
- 性能面板：显示 FPS、渲染点数、源点数、抽样比例、GPU/WebGL 能力、Worker 队列和内存估算。

这一阶段仍然以 LAS 为输入。可以生成浏览器端临时索引或项目缓存，但不需要支持 LAZ。

### 阶段 5：电力巡检业务能力

目标是从点云查看器升级为巡检辅助工作台。

- 输电专题：导线/地线/杆塔/绝缘子/植被/交跨的专题显示、筛选和统计。
- 变电专题：设备、构架、导线、绝缘子、地面和建筑物的分类显示与局部核查。
- 净空分析增强：从当前抽样近似升级为线路分段、导线拟合、风险点聚类和结果复核。
- 航线编辑增强：航点拖拽、批量调整、姿态联动、目标点编辑、拍摄角度验证和航线 QA 检查。
- 问题点管理：用户可添加风险点、截图、备注、类别、处理状态，并导出 JSON 或报告。
- 巡检结果导出：截图、剖面、净空风险、航点清单、部件点清单和 QA 结果统一导出。

### 阶段 6：工程化拆分与可维护性

目标是避免 `main.js` 继续膨胀，让后续功能可持续开发。

- 状态层拆分：项目、点云图层、航线、分析、UI 设置分别管理。
- 渲染层拆分：点云 renderer、航线 renderer、标签 overlay、编辑 gizmo、拾取服务各自独立。
- 数据层拆分：LAS reader、项目坐标系、route adapter、分析模型和导出模型分离。
- UI 组件化：工具栏、项目树、属性面板、表格、弹窗、状态栏形成可复用组件。
- 测试增强：LAS 解析、路线解析、颜色表、坐标转换、分析算法、交互状态和浏览器 smoke test 都纳入验证。

### 阶段 7：WASM/C++ 复用评估

目标是把已有 C++ 算法资产以可控方式引入 Web。

- 优先评估计算密集模块：空间索引、近邻搜索、导线拟合、净空分析、分类统计。
- C++ 核心应先整理为不依赖 Qt UI 的纯算法库，再编译为 WASM。
- WASM 只承担计算核心，UI、状态、文件选择和 WebGL 渲染仍保留 Web 架构。
- 只有当 JavaScript/Worker 难以满足性能或已有 C++ 算法复用价值明确时，才进入这一阶段。

### 阶段 8：WebGPU 评估

目标是在 WebGL 遇到明确瓶颈后，再引入更现代的 GPU 管线。

- WebGL 仍作为主线，因为兼容性更稳定。
- 当多 buffer 管理、GPU picking、GPU 计算、超大点预算或复杂后处理成为核心瓶颈后，再评估 WebGPU。
- WebGPU 可用于更大规模 buffer 管理、GPU 计算、可视化效果和现代渲染管线。
- WebGPU 不应早期引入，避免在业务工作流尚未稳定时增加平台风险。

## 验收指标

每个阶段都需要可验证的指标，而不是只看功能是否“做了”。

- 浏览：空闲 CPU/GPU 占用低，旋转/平移/缩放没有明显卡顿，快捷视角和聚焦稳定。
- 加载：大 LAS 有进度反馈，首屏尽快出现粗略点云，加载失败有可理解原因。
- 渲染：可见点数、抽样比例、EDL、点大小、分类颜色和标签层级可控。
- 编辑：航点、部件点、视椎体、右下角相机预览、属性面板和表格选中保持同步。
- 分析：测量、剖面、净空结果可复核，可定位到场景，可导出。
- UI：主要功能不依赖长滚动寻找，按钮、表格、弹窗、提示和颜色体系一致。
- 稳定性：`npm run validate`、LAS 解析测试、航线 JSON 测试、分析测试和浏览器 smoke test 持续通过。

## 推荐近期优先级

1. 先做 V0.2 和 V0.3，把浏览、交互、编辑和 UI 工作台体验补齐。
2. 再做 V0.4，把 LAS 解析、颜色重建和分析重计算移入 Worker。
3. 然后做 V0.5，解决多数据、大场景和精细浏览的性能问题。
4. 最后做 V0.6 及之后，把电力巡检业务分析、WASM/C++ 复用和 WebGPU 逐步接入。

这条路线的核心原则是：先让用户愿意每天使用，再让它承载更大的数据；先把 LAS、航线和巡检流程做扎实，再投入重型渲染和算法架构。
